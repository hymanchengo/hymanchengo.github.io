# 条例9： 较try-finally优先使用try-with-resources

Java库包含了许多必须调用close方法来手动关闭的资源。包括InputStream,OutputStream和java.sql.Connection等示例。 关闭资源通常被客户代码所忽视，带来可怕的性能后果。尽管许多资源使用finalizers作为安全网，但finalize并不太奏效。

一直一来，try-finally语句是确保资源被合适释放的最好方式，甚至在遇到异常或返回语句时：

// try-finally - 不再是最好的关闭资源方式
static String firstLineOfFile(String path) throws IOException {
  BufferedReader br = new BufferedReader(new FileReader(path));
  try {
    return br.readLine();
  } finally {
    br.close();
  }
}

这看起来并不坏，但是当添加第二段资源时，事情变得有点糟。

// 当使用一个以上的资源时，try-finally是丑陋的
static void copy(String src, String dst) throws IOException {
  InputStream in =new FileInputStream(src);
  try {
    OutputStream out = new FileOutputStream(dst);
    try {
      byte[] buf = new byte[BUFFER_SIZE];
      int n;
      while ((n = in.read(buf)) >= 0)
      out.write(buf, 0, n);
    } finally {
      out.close();
    }
  } finally { in .close();
  }
}

也许难以置信，甚至优秀的程序员在大多数时候都能搞错。最初，我在Java Puzzlers第88页弄错过一次，许多年都没人注意到它。 事实上，2007年Java库中的close方法的三分之二的使用是错误的。

如前面2段代码所示。即便使用try-finally语句关闭资源的正确代码，也存在细微的缺陷。try块和finally块中的代码都能扔出异常。 比如，在firstLineOfFile方法中，调用readline可能因为底层的物理设备的故障而扔出异常。调用close也可能因相同的原因失败。 在这些情况下，第二个异常完全消除了第一个异常。异常堆栈里不会出现第一个异常的记录，在真实的系统中将极大的复杂化调试—— 通常为了诊断某个问题，你希望看到出现的第一个异常。通常为了看到第一个异常可以编写代码来压制第二个异常，但是实际上没人这么做， 因为它太冗长了。

当Java 7引入了try-with-resources语句后所有这些问题都一口气解决了。为了能使用这个构造，资源必须实现AutoClosable接口， 该接口只包含一个返回void的close方法。Java库和第三方类库的许多类和接口目前实现或继承了AutoCloseable。如果编写一个必须 被关闭的资源类，这个类应该也要实现AutoCloseable接口。

如下为我们使用try-with-resources的第一个示例：

// try-with-resources - 关闭资源的最好方式
static String firstLineOfFile(String path) throws IOException {
  try (BufferedReader br = new BufferedReader(
  new FileReader(path))) {
    return br.readLine();
  }
}

下面为我们使用try-with-resources的第二个示例：

// try-with-resources 用在多个资源上 简短而甜美
static void copy(String src, String dst) throws IOException {
  try (InputStream in =new FileInputStream(src);
  OutputStream out = new FileOutputStream(dst)) {
    byte[] buf = new byte[BUFFER_SIZE];
    int n;
    while ((n = in.read(buf)) >= 0)
    out.write(buf, 0, n);
  }
}

和之前的代码比使用了try-with-resources的版本更简短，更具可读性且它们提供了更好的诊断。如果firstLineOfFile方法 中的readLine调用和(隐形的)close方法调用都扔出异常，close方法扔出的异常会被压制，只会扔出调用readLine的异常。 事实上，可以压制多个异常以保留你确实希望看到的异常。这些被压制的异常不是单纯地被丢弃，它们被打印在堆栈中，带有一个 已被压制的符号。也可以通过getSuppressed方法来以编程的形式访问这些被压制的异常。getSuppressed方法是在Java7中 被添加到Throwable中的。

你可以把catch语句放在try-with=resources语句上，就像try-finally语句那样。这允许你处理异常而无需用另一层嵌套弄乱代码。 举一个稍作的例子，下面版本firstLineOfFile方法不扔出异常，但是接受一个默认值以在不能打开或读取文件时返回它。

// 带有catch子句的 try-with-resources
static String firstLineOfFile(String path, String defaultVal) {
try (BufferedReader br = new BufferedReader(
new FileReader(path))) {
return br.readLine();
} catch (IOException e) {
return defaultVal;
}
}

本条例主旨明确：当处理必须关闭的资源时，相对于try-finally总是优先使用try-with-resources。这样代码 更加简短清楚，它产生的异常更加有用。try-with-resource语句使得编写正确的代码以使用必须被关闭的资源时 更加容易，而使用try-finally是绝无实际可能做到这点的。
