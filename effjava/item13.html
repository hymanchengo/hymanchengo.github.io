# 条例13：明智地重写clone方法

Cloneable接口作为混入接口（条例20）指示类允许克隆。不幸的是，它不能达到这个目的。它的主要缺陷是没有clone方法。且Object类的clone方法 是受保护的。如果不使用反射（条例65），你不能仅仅因为类实现了Cloneable接口，就调用对象上的clone方法。甚至反射调用可能失败，因为不能保证 对象有可访问的clone方法。尽管存在这个缺点和其他许多缺点，这个工具还是得到了广泛的使用，所以有必要了解它。这个条例告诉你怎么实现运作良好的 clone方法，讨论什么时候适合做这件事，且呈现可选项。

既然没有方法，那么Cloneable做什么的呢？它确定了Object的受保护clone方法实现的行为：如果了实现了Cloneable，那么Object的clone方法返回对象的 逐字段拷贝。否则扔出CloneNotSupportedException。这是接口非常不典型的用法，不该被模仿。正常情况下，实现一个接口意味着一个类可以为它的客户做 些什么。而在实现Cloneable接口的例子中，意味着修改父类受保护方法的行为。

尽管规范没有提到，实际上，实现Cloneable接口的类希望提供一个功能合理的公共clone方法。为了达成这点，类和它所有的父类必须遵守一个复杂的不强制的， 缺少正式记录的协议。导致的机制是脆落的，危险的，及语言外的：不需要调用构造函数创建对象。

clone方法的通用契约很弱。如下，从Object规范中复制的： 创建并返回对象的副本。复制的准确含义可能取决于对象的类。通用的目的是对于任何对象x，表达式x.clone() != x是返回true的，表达式x.clone().getClass() == x.getClass() 返回true，但是这些不是绝对的要求。尽管通常x.clone().equals(x)返回true，但这也不是绝对要求。按照约定，这个方法返回的对象应该通过调用super.clone来获取。如果类和它 所有的父类（除了Object类）遵循这个约定，那么将会是这种情况：x.clone().getClass() == x.getClass。按照约定，返回的对象应该独立于被克隆的对象。要达到这个独立性，可能 需要在返回super.clone返回的对象前需要修改一个或多个字段。

这种机制大概类似于构造函数链，除了它不是强制的这一点外：如果类的clone方法返回的实例不是通过调用super.clone方法而是调用构造函数获取的。编译器不会提示， 但是如果那个类的子类调用super.clone方法，导致的对象类型会错误，阻止子类的clone方法正常运行。如果重写了clone的类是final的，这个约定可以安全的忽略，因为 因为不用考虑子类。但是如果final类的clone方法不调用super.clone，那么这个类没有理由实现Cloneable接口，因为它不依赖Object的clone实现的行为。

假设你想在其父类提供了行为良好的clone方法的类中实现Cloneable接口，首先调用super.clone。你取回的对象将是原对象的功能完整的复本。类中任何字段将具有和原对象 的对应字段值相等。如果每个字段包含了原始值或者不可变对象的引用，那么返回的对象可能正是你所需要的，在这种情况下不需要进一步处理。这是条例11中的PhoneNumber类的例子。， 但是注意不可变类永远不该提供clone方法因为这仅仅是无意义拷贝。了解这个注意点后，下面是PhoneNumber类的clone方法看上去的样子:

//类的克隆方法不引用可变状态
@Override
public PhoneNumber clone() {
  try {
    return (PhoneNumber) super.clone();
  } catch(CloneNotSupportedException e) {
    throw new AssertionError(); //不可能发生
  }
}

为了使如上clone方法工作，PhoneNumber的类声明应该被修改为实现Cloneable。尽管Object的clone方法返回Object，这个clone方法返回PhoneNumber。 这样做合法且合乎需要，因为Java支持协变返回类型。换句话说，重写方法的返回类型可以是被重写的方法的返回类型的子类型。这消除了在客户端转换的需要。 我们必须将Object中的super.clone的结果转换成PhoneNumber然后返回它，但是转换时确保可以成功的。

super.clone的调用包含在try-catch块中。这是因为Object声明了clone方法扔出CloneNotSupportedException，这是一个检查异常。因为PhoneNumber实现了 Cloneable接口，我们知道调用super.clone会成功。需要此样板表名CloneNotSupportedException应该是未检查的（条例71）

如果对象包含引用了可变对象的字段，前面展示的clone的简单实现可能是灾难性的。比如，考虑条例7中的栈：

public class Stack {
  private Object[] elements;
  private int size = 0;
  private static final int DEFAULT_INITIAL_CAPACITY = 16;
  public Stack() {
    this.elements = new Object[DEFAULT_INITIAL_CAPACITY];
  }
  public void push(Object e) {
    ensureCapacity();
    elements[size++] = e;
  }
  public Object pop() {
    if (size == 0) throw new EmptyStackException();
    Object result = elements[--size];
    elements[size] = null; // Eliminate obsolete reference
    return result;
  }
  // Ensure space for at least one more element.
  private void ensureCapacity() {
    if (elements.length == size) elements = Arrays.copyOf(elements, 2 * size + 1);
  }
}

假设你想要使这个类可以克隆。如果clone方法只是返回super.clone()，导致的Stack实例在它的size字段会有正确的值，但是它的elements字段 将会引用原先栈实例的相同数组。修改源将摧毁克隆中的不变性，反之修改克隆中的也会摧毁源的不变性。你将很快发现你的程序产生无意义的结果或 扔出NullPointerException。

这个场景永远也不会作为调用Stack类中单个构造函数的结果出现。实际上，clone方法功能类似构造函数；你必须确保它对原始的对象没有损害。 它在clone对象上建立起不变性。为了让Stack的clone方法能正常运作，必须拷贝栈的内部。最简单的方式是在elements数组上递归调用clone。

// 类的clone方法引用了可变状态
@Override 
public Stack clone() {
  try {
    Stack result = (Stack) super.clone();
    result.elements = elements.clone();
    return result;
  } catch(CloneNotSupportedException e) {
    throw new AssertionError();
  }
}

注意我们没有必须将elements.clone的结果转换为Object[]。在数组上调用clone返回的数组的运行时和编译器的类型和被克隆的数组的类型相同。 这是复制数组首选的惯用方法。事实上，数组是克隆设施唯一引人注目的用途。

也要注意下，如果elements字段是final的，那么早先的解决方法将会不起作用，因为clone被禁止将新的值赋给这个字段。这是基本的问题： 与序列化一样，Cloneable体系结果和引用可变独享的final字段的正常使用比较兼容，除了在可变对象可以在对象和它的克隆体之间安全共享。 为了让一个类可克隆，也许像需要从一些字段移除final修饰符。

递归调用clone不总是足够的。比如，假设你为内容由桶数组构成的哈希表编写clone方法，哈希桶中的每一个元素已禁用了键值对链表的第一个条目。 出于性能，类实现它自身轻量级单链表，内部不是使用java.util.LinkedList：

public class HashTable implements Cloneable {
  private Entry[] buckets = ...;
  private static class Entry {
    final Object key;
    Object value;
    Entry next;
    Entry(Object key, Object value, Entry next) {
      this.key = key;
      this.value = value;
      this.next = next;
    }
  }... // Remainder omitted
}

假设你只递归克隆了桶数组，正如我们之前在Stack中做的：

// 裂开的克隆方法 - 导致共享可变状态
@Override 
public HashTable clone() {
  try {
    HashTable result = (HashTable) super.clone();
    result.buckets = buckets.clone();
    return result;
  } catch(CloneNotSupportedException e) {
    throw new AssertionError();
  }
}

尽管克隆体具有自己的桶数组，数组和源引用相同的链表，这容易在克隆体和源中导致不确定的行为。要修复这个问题，需要拷贝组成每个桶的链表。 下面是一个常用方式

//复杂可变状态类的clone方法
public class HashTable implements Cloneable {
  private Entry[] buckets = ...;
  private static class Entry {
    final Object key;
    Object value;
    Entry next;
    Entry(Object key, Object value, Entry next) {
      this.key = key;
      this.value = value;
      this.next = next;
    }
    // 递归拷贝这个条目开头的链表
    Entry deepCopy() {
      return new Entry(key, value, next == null ? null: next.deepCopy());
    }
  }
   @Override 
   public HashTable clone() {
    try {
      HashTable result = (HashTable) super.clone();
      result.buckets = new Entry[buckets.length];
      for (int i = 0; i < buckets.length; i++)
      if (buckets[i] != null) result.buckets[i] = buckets[i].deepCopy();
      return result;
    } catch(CloneNotSupportedException e) {
      throw new AssertionError();
    }
  }... // Remainder omitted
}

增加私有类HashTable.Entry来支持深拷贝方法。HashTable上的clone方法分配一个合适大小的新的桶数组，然后迭代源桶数组，深拷贝每个 非空的桶。Entry上的deepCopy方法递归调用自身拷贝由这个条目开头的这个链表。尽管这个技术聪明且在桶不是很长的情况下工作良好， 但它不是克隆链表的好的方式，因为它为链表的每个元素消耗一个栈帧。如果链表长，这可能容易导致栈溢出。要阻止栈溢出发生，可以用 迭代来替换deepCopy中的递归：

//迭代拷贝拷贝由这个条目开头的链表
Entry deepCopy() {
  Entry result = new Entry(key, value, next);
  for (Entry p = result; p.next != null; p = p.next)
  p.next = new Entry(p.next.key, p.next.value, p.next.next);
  return result;
}

克隆复杂可变对象的最后一种方式是调用super.clone，设置结果对象中的所有状态为初始状态，然后调用高层方法重新生成源对象的状态。 在我们的HashTable例子中，buckets字段会被初始化为新的bucket数组，put(key,value)方法（没有展示）由正在被克隆的哈希表中的每个键值 映射调用。这种方式通常生成一个简单的非常优雅的方法，它不像直接调用clone内部方法那样运行快。尽管这种方式很干净，但它和整个克隆体系 相对立，因为它盲目覆盖了构成克隆体系结构基础的逐字段的对象副本。

像构造函数一样，clone方法永远不能在正在构造中的克隆体上调用一个重写的方法。（条例19）。如果clone调用了在子类中重写的方法，那么 这个方法将会在子类获得机会在克隆中完成状态前执行，很可能导致克隆和源的损坏。所以在前面讨论的put(key,value)方法方法应该是final或 private。（如果是private的，那么它可能是非final公共方法的helper方法）

Object的clone方法声明为扔出CloneNotSupportedException，但是重写方法不需要。公共的clone方法应该省略throws语句。因为不扔出检查 异常的方法更容易使用（条例71）。

当设计一个用于继承（条例19）的类时有两个选择，但是不管你选择哪个，类不该实现Cloneable。你可以模仿Object的行为，实现合适功能的受保护的 clone方法，且声明扔出CloneNotSupportedException。这给子类是否实现Cloneable的自由，就好像他们直接继承了Object。可选的，你可选择不实现 工作的clone方法，且阻止子类实现，通过提供如下降级的clone实现：

// 用于扩展类的clone方法不支持Cloneable
@Override
protected final Object clone() throws CloneNotSupportedException {
    throw new CloneNotSupportedException();
}

还有一个细节值得注意。如果你编写了实现Cloneable的线程安全类，记得它的clone方法必须适当地同步。就像其他方法。Object的clone方法 不是同步的，所以即便它的实现令人满意，你也要写一个同步的clone方法返回super.clone()。

回顾下，所有实现了Cloneable的类应该重写clone方法，它是返回类型为类自身的公共方法。这个方法中先要调用super.clone，然后需要调整的字段 再逐个调整。通常，这意味着拷贝组成对象内部深层结构的可变对象，将克隆体对这些对象的引用替换为对这些对象副本的引用。

尽管这些内部副本通常可以通过递归调用clone方法制作，但这通常不是最好的方式。如果类只包含原始类型字段或对不可变对象的引用，那么可能就不需要 做任何调整。这个规则有例外情况。比如，代表序号或其他唯一ID的字段需要被调整即便它是原始类型或不可变的。

真的值得这么复杂吗？很少。如果你继承了已经实现Cloneable的类，你没得选只能实现良好行为的clone方法。否则，通常最好提供对象拷贝的替代方案。 一个较好的对象拷贝的方式是提供拷贝构造函数或拷贝工厂。拷贝构造函数仅仅是一个接收单个类型是包含构造函数自身的类的参数，比如：

// 拷贝构造函数
public Yum(Yum yum) { ... };

拷贝工厂是拷贝构造函数的静态工厂类似方法

// 拷贝工厂
public static Yum newInstance(Yum yum) { ... };

拷贝构造函数方式和他的静态工厂变体比Cloneable/clone具有更多的优点：他们不依赖语言外的有风险的对象创建机制。他们不要求非强制遵循 较少文档记录的惯例。他们不和合适使用的final字段冲突。他们不扔出没必要的检查检查。他们不需要转换。

而且，拷贝构造函数或工厂可以接受类型是该类实现的接口的参数。比如，根据惯例，所有通过集合实现提供了参数是类型Collection或Map的 构造函数。基于接口的拷贝构造函数和工厂，更多地被称为转换构造函数和转换工厂，允许客户端代码选择拷贝的实现类型而不是强制够护短 接受源的实现类型。比如，假设有一个HashSet，s，然后你想将其拷贝为一个TreeSet。clone方法不能提供这个功能。但是使用转换构造函数 new TreeSet<>(s)则很容易。

考虑Cloneable关乎的所有问题，新的接口不应该继承它，新的继承类不该实现它。尽管对于final类实现Cloneable是无害的，但是这应该被视为 性能优化，保留给极少数有正当理由的地方使用。（条例67）作为规则，拷贝功能最好由构造函数或工厂提供。这个规则不可忽视的例外是数组， 最好使用克隆方法拷贝数组。
