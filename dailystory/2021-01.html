# 2021-1-19

  • 真实部署前最好的方式是在相同或者逼近的环境中进行演练。
    • 可以知道要准备哪些包和配置，知道步骤
    • 可以事先摸清可能遇到的坑并填掉
    • 真实部署可以更有效率

本次在windows部署nginx + spring boot + redis + mysql技术栈，聊一聊大致步骤

  • 首先我画了前后端分离的部署图。理清需要部署的组件。
  • 下载准备目标环境支持的nginx,java,redis,mysql组件
  • 安装时，主要注意的是安装为windows服务，使用winsw (opens new window)这个小工具（需要运行在.net framework上）
  • 为了逼近真实环境，我在本机虚拟了windows 7。这里主要学到的是主机和虚拟机使用NAT网络，然后解决互相访问的问题。主机使用端口转发 ，虚拟机使用主机IP即可互相访问。
  • 本次部署，学到的一点是部署需要靠自动化或半自动化。这次体验了下windows批处理脚本的编写，部署时效率提高了不少。
  • 其次比较重点的就是nginx对前端和后端服务的配置，mysql schema提前导出（dump）及导入（source），一些初始化脚本，比如admin用户信息配置等

总言之，自动化可以做得更好。

  • 补充昨天，同事提出的使用灰度来区分，一套研发环境切换pg和oracle的问题 背景是我们使用了Apollo,而且项目代码支持pg和oracle 2套环境。研发每次开发测试不同db时需要修改Apollo配置，一种是直接在apollo portal里改， 这样的问题是容易导致混乱，因为不是你一个人用研发的配置。还有一种是更改本地的apollo配置，这样不影响他人。但是改来改去也很繁琐。
    • 之前有想做个工具来切换本地配置，思路是只需要第一次准备2个配置文件对应2套环境，要使用哪一套时，点击切换，实质上就是文件拷贝。
    • 同事提出用apollo中的灰度来实现差异配置，将一套环境配置到灰度中，然后要用的人添加自己的ip到灰度中。这个方法也巧妙地解决了问题
      • 我顺水推舟，让好东西和团队分享。这样团队受益，也共同遵守规范，以免引起混乱
    • 事后来看，其实Apollo应该也是提供了些API来操作的。扩展下来，可做的事情也挺多的。这个痛点完全可以更加自动化掉。

可能你会好奇，为啥不建多个环境，是啊，为什么不呢？

记使用sql多年，但对许多概念还是停留在浅显了解的层面。虽使用过oracle, sql server, mysql, postgres 但很多知识都是在使用时到网上搜索， 并不成体系，没有沉淀。这些在学习postgres时自己从0到1建库，建表等一步步去学习和理解那些常用的知识点。一来构建大致的体系，二来，按照 官方手册来进行知识理解和整理，感觉和从网上了解的只言片语，确实非常不一样，另外通过自己实际操作来理解这些知识点，又增强了对模棱两可的知识 的理解。比如主键等其实是一种约束，在以往的认知里就知道主键，不能重复，且唯一，但完全忽略了约束的概念。另外创建唯一约束和主键约束时会创建 索引，因为唯一约束是通过唯一索引来实施的，再比如连接分为cross连接，内连接和外连接，外连接又分为左连接，右连接及全连接。以前也是了解这些 概念的，但是不能讲出来。说明理解还不透彻，现在以实例来操作一番，确实更清楚。当然数据库的深入学习还需要大量的实操。这只是开始，看了再多， 确实不如自己操作一遍好啊。

  • 记一次问题排查

oracle库里创建主键约束，发现自动创建的索引字段顺序没有按预期的那样改变。最后还原出来是发现：原先之前建了这个同名索引，再建约束时，它并没有 再建索引，而是关联了这个索引。所以出现键中的字段顺序和索引中的字段顺序不一致。解决方法是先删了索引，再创建约束。或删除这个创建的约束的同时进行 级联索引删除，再创建约束。
