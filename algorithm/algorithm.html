# 数组

数组为什么从0开始编号，而不是更符合人类直觉的1？

数组特性

  • 线性数据结构
  • 连续的内存空间
  • 存储相同类型的数据
  • 随机访问
  • 插入，删除低效
    • 根据考虑插入位置，探讨
    • 尾插/删：最好时间复杂度O(1)
    • 头插/删：最坏时间复杂度O(n)
    • 平均时间复杂度(1+..+n)/n = O(n)
    • 在特殊场景下，可以优化插入，如在k位置插入一个新元素，可以将新元素插入k位置，而k位置原先的元素直接移到数组末尾。这样复杂度为O(1)
    • 在特殊场景下，可以优化删除，标记删除元素，而不立即删除，等数据大小不够用时触发删除，这样避免频繁移动元素。这样复杂度为O(1)。这和jvm gc Mark-Sweep算法核心思想一致。

数组为什么可以随机访问？ 因为连续的内存空间和相同类型数据这2个特性使得任何一个数组元素的存储地址可以被直接计算出，因此可以直接访问。 因此第n个元素存储地址的计算：初始地址 + (n * 1个数据元素占据地址范围)。第一个元素地址就是初始地址，因此n取0。如果n从1开始编号，则计算每个元素存储地址的时候需要额外进行n-1的计算。

数组相关时间复杂度

  • 随机访问O(1)

C语言数组越界问题 由于C语言直接访问内存，下面示例数组越界访问arr[3]，如果这个内存存放的是i，那么i又被置为0，戏剧的事情发生了，就会一直循环下去。

int main(int argc, char* argv[]){
    int i = 0;
    int arr[3] = {0}; //arr长度为3
    for(; i<=3; i++){ // i 从0到3。3的时候越界
        arr[i] = 0; 
        printf("hello world\n");
    }
    return 0;
}

Java ArrayList与数组 Java中的ArrayList是数组实现的。它的优点是提供了丰富的操作数组的API，使用者无需关系数组越界，容量不够的问题。容量不够时会自动进行扩容。
数组相较于ArrayList的使用场景 - 性能要求高，直接使用数组 - 基础数据类型，使用数组 - 简单数组操作

Java中数组为什么限制大小为private static final int MAX_ARRAY_SIZE = Integer.MAX_VALUE - 8; 通过length属性可以看到定义为int。那为什么不定义long？ Integer.MAX_VALUE为2^31，约2G。我们pc内存常见有4G,8G,16G等。如果用long表示数组大小，那可表示的内存大小远远超过我们现在使用的内存，完全没有必要。

# 链表

经典使用场景：LRU缓存淘汰算法

缓存：提高数据读取性能的常用技术，常见的有CPU 1,2,3级缓存。内存缓存。浏览器缓存。

缓存通常都不会太大，因此当缓存满了，按照什么策略进行清理呢？常见的有先进先出FIFO,最少使用LFU,最近最少使用LRU。 类比对衣橱里的衣服进行整理，为新买衣服腾空间，最常见的就是把最早买的那批先打包，因为旧了。其次对不怎么想穿的进行打包，最后最近不怎么用到的打包，比如冬天了，夏天的衣服就先打包。

链表vs数组

  • 内存利用率 数组需要连续存储空间，而链表则不需要，链表用指针串起元素结点，那么链表较数组可以利用碎片内存空间，内存利用率高。
  • 性能
    • 数组查找性能高O(1)，链表查找性能低O(n)
    • 数组插入、删除性能低O(n)，链表插入删除性能高O(1)
    • 因为数组是连续的内存空间，对于CPU缓存较为友好，因为CPU缓存可以轻易读入一段连续空间。而链表因为内存空间不连续，则CPU一次读取，可能不能加载整个链表（寻址）。
  • 动态扩展性
    • 创建新数组需要定义大小，如果事先知道所需的尺寸，按需申请是没问题，一旦不能或错误估计数组大小，那么要么空间申请大了浪费，要么申请小了，需要动态扩容并拷贝，效率低。链表的结构则更为灵活， 天然支持动态扩展。
  • 内存碎片
    • 链表频繁插入，删除，可能导致更多的内存碎片，需要更频繁的gc。 链表术语 结点：链表上的每个内存块，包括数据和指针两部分 后继指针next：结点中指向下个内存块地址的部分

链表种类

  • 单链表
    • 不同节点个数的链表
      • 0个结点：空链表
      • 1个结点：结点指针不指向任何其他结点，指针为NULL。1个结点既是头结点，也是尾结点。头结点记录链表基地址
      • 2个结点：头结点指向尾结点
      • 3个以上节点：头结点只有后继结点，中间结点有前驱和后继节点，尾结点只有前驱结点
    • 特性
      • 插入/删除
        • 时间复杂度O(1)，(这边的时间复杂度不包括查找的复杂度) 在x->z链表中插入y, x->y, y->z，在x->y->z链表中删除y,x->z, y->null
      • 访问
        • 时间复杂度O(n)
  • 循环链表
    • 原先单链表的尾结点指向头结点形成了循环链表，循环链表每个结点都有前驱和后继结点。
    • 约瑟夫问题
  • 双向链表
    • 每个结点有两个指针，分别指向前驱结点和后继结点，因此比存储相同数据的单链表更加占用空间。
    • 可以按两个方向遍历链表
      • 给定结点，删除（插入）其前驱结点，双向链表通过前驱指针即可定位到前驱结点，时间复杂度O(1)，而单链表则需从链表头部开始查找，时间复杂度O(n)
      • 给定值，删除（插入）其前驱，则单链表和双向链表则要从头遍历找出值等于给定值的节点进行操作，时间复杂度都为O(n)
    • 体现了空间换时间的思想
      • 缓存也体现了空间换时间
  • 双向循环链表

# 队列

线程池设定多大合适？是不是越大越好？ 是不是线程数越多，任务处理越快？这不是简单的线性关系，因为受CPU资源的限制。我们可以想象，在CPU资源空闲时间，任务处理速度和线程数呈线性关系。当CPU占用率达到一定的数值后，线性关系便不能满足， 而且由于线程数目的增加，导致CPU频繁切换。性能下降。 因此线程池的大小需要综合考虑处理的任务特点和硬件环境来事先设置。 队列在生活中经常看到，如典型的排队打饭场景。 队列特性：先进先出。 队列也是操作受限的数据结构，队尾入队，队头出队。 队列分类

  • 顺序队列 用数组实现的队列 用2个指针，1个执行队头，一个指向队尾。当出队时移动队头指针，当入队时移动队尾指针，当队尾指针移到数组最大长度时，将没有空间，此时可以触发一下数据搬移，将队头到队尾的这部分元素搬移到数组开始后连续的 位置，这样出队后入队的时间复杂度都是O(1)（出队的时间复杂度用均摊法计算出来为O(1)）

  • 链式队列 用链表实现的队列。依然使用2个指针。

  • 循环队列。考虑队空（head = tail），队满的判定条件(tail + 1) % n = head 解决顺序队列需要搬移数据的问题。 特性队列

  • 阻塞队列

    • 队头无元素可取，阻塞
    • 队尾无位置可入队，阻塞
    • 阻塞队列实现生产者消费者模型。
      • 消费者从队头取数据，没有则阻塞等待。
      • 生产者从队尾放数据，没有位置则等待。说明消费不过来。
      • 使用n个消费者，1个生产者的模式提示数据处理效率。

  • 并发队列

    • 支持多线程环境操作的队列。
    • 线程安全的队列
    • 实现方式
      • 入队，出队加锁。粒度太大
      • 基于数组的循环队列，利用CAS操作 对于资源优先的池，如线程池，数据库连接池，当无可用资源时，陆续到来的请求可以使用队列来实现请求排队，对于链表实现的队列是无界队列，请求队列可能相对较长，导致响应时间长， 对于数组实现的队列，是有界队列，当排队的请求塞满队列时，后面的请求可以直接拒绝。

# 数据结构与算法关系

  • 相辅相成，算法作用于特定的数据结构。也依托于特定数据结构（如二分查找依托数组的随机访问特性，而不能作用于链表上）。
  • 不同数据结构产生的算法也不同。
  • 数据结构可以类比为存储，算法类比为计算。
  • 数据结构是静态的，算法是盘活数据结构的关键。
  • 数据结构提供数据存储的方式，算法提供操作数据的方式。

# 复杂度分析

衡量算法执行效率的指标：时间、空间复杂度分析
why复杂度分析？

  • 事后统计法的局限
    • 测试结果依赖测试环境，如代码在i3和i9上跑的结果比对
    • 测试结果受规模影响大。如待排序数据的规模可能对评估算法的优劣产生不同的结果。另外排序数据有序性对算法评估也有影响。 因此，基于事后统计法，即在实际机器上跑一下代码统计时空消耗的方法有诸多局限，我们需要粗略估计算法执行效率的方法
  • 大O复杂度表示法（或渐进时间复杂度）（简称时间复杂度）
    给定如下代码，假定每行代码执行耗时一样（记作u），那么耗时2nu + 2u，总执行时间T(n)与每行代码执行次数成正比。
 int cal(int n) {
   int sum = 0; // 1u
   int i = 1;   // 1u
   for (; i <= n; ++i) { // nu
     sum = sum + i;     // nu
   }
   return sum;
 }

根据上面分析，下面这段代码执行耗时(3 + n + n + n^2 + n^2)u = (2n^2 + 2n + 3)u


 int cal(int n) {
   int sum = 0;
   int i = 1;
   int j = 1;
   for (; i <= n; ++i) {
     j = 1;
     for (; j <= n; ++j) {
       sum = sum +  i * j;
     }
   }
 }

从上面2段代码总结规律：所有代码的执行时间T(n)与每行代码执行次数f(n)成正比。T(n) = O(f(n))。O表示正比关系。 因此上面2段代码耗时用公式分别表示为T(n) = O(2n + 2), T(n) = O(2n^2 + 2n + 3)。进一步，如果n足够大，低阶、常量、系数等不足以影响复杂度的整个走势，因此上面2段代码 时间复杂度可进一步表示我T(n) = O(n), T(n) = O(n^2)。
大O时间复杂度并不具体表示代码真正执行时间，而是表示代码执行时间随数据规模增长的变化趋势。

  • 时间复杂度分析的实用方法
    • 只关注循环次数执行最多的一段代码
    • 加法法则T(n)=T1(n)+T2(n)=max(O(f(n)), O(g(n))) =O(max(f(n), g(n)))。需要注意不管常量循环的次数有多大，它不反应增长趋势，即便常量1000>n^2，复杂度依然取n^2
      • 当有2个数据规模的循环时，如m和n,因为不知道哪个数量级大，不能随意取其中一个，因此加法的算法复杂度表示为O(m + n)
    • 乘法法则：嵌套代码的复杂度等于嵌套内外代码复杂度的乘积。T(n) = T1(n) * T2(n) = O(f(n)) * O(g(n)) = O(f(n) * g(n))
  • 常见的时间复杂度量级
    • 多项式量级
      • 常量阶O(1)
        • 常量级时间复杂度
        • 表示代码执行时间不随n的增大而增长的时间复杂度
        • 若代码中不存在循环、递归等，即便上千上万行代码依然是O(1)复杂度
      • 对数阶O(logn) 这边很多人可能有疑问这个logn是以多少为底的，实际上并不重要。统一记作logn。因为对数间可以转换如log3(n) = log3(2) * log2(n)，而log3(2)是个常数，可以忽略。因此以对数阶复杂度
        我们直接忽略底数，记作O(logn)
      • 线性阶O(n)
      • 线性对数阶O(nlogn)
        • 理解O(nlogn)，可以想象外层n次循环，内层的时间复杂度是O(logn)，这样总复杂度就是O(nlogn)
        • 示例
          • 归并排序
          • 快速排序
      • 平方阶O(n^2)
      • 立方阶O(n^3)
      • k次方阶O(n^k)
    • 非多项式量级 非多项式量级的算法问题成为NP（None-Deterministic Polynomial非确定多项式）问题。随着n的增大，算法执行时间急剧增加，算法效率极其低下。
      • 指数阶O(2^n)
      • 阶乘阶O(n!)
  • 空间复杂度（渐近空间复杂度） 表示算法的存储空间与数据规模的增长关系 常用空间复杂度O(1),O(n),O(n^2)
  • 考量几个典型的算法复杂度 考虑如下代码
// n表示数组array的长度, 返回无序数组中指定元素的位置
int find(int[] array, int n, int x) {
  int i = 0;
  int pos = -1;
  for (; i < n; ++i) {
    if (array[i] == x) {
        pos = i;
        break;
    }
  }
  return pos;
}

正常情况不需要区分最好，最坏，平均时间复杂度。只有存在量级差距时才进行区分。

  • 最好情况时间复杂度 最理想情况下，元素x出现在数组中的第一个位置处：O(1)
  • 最坏情况时间复杂度 最糟糕情况下，元素x不存在于数组中：O(n)
  • 平均情况时间复杂度（加权平均时间复杂度） 考虑最好情况和最坏情况出现的概率不大，因此引入平均情况时间复杂度： 不考虑概率：(1（在第一个位置） + 2 + ... + n + n（不在数组中）) / (n + 1) = (n^2 + 3n ) /2(n + 1)，简化后为O(n) 考虑概率：每个位置出现的概率为1/2n。那么1*1/2n + 2*1/2n+...+ n*1/2n + n*1/2=(3n+1)/4，该值为加权平均值（期望值），复杂度依然是O(n)
  • 均摊时间复杂度 用另一段代码理解均摊时间复杂度。这段代码用概率论可以分析，不用概率率也可以，因为它是有规律的。对于前n次插入复杂度是O(1)，对于n+1次复杂度为O(n). 因此这一次的复杂度可以均摊到前n次中。因此整个复杂度就是O(1)。
      // array表示一个长度为n的数组
  // 代码中的array.length就等于n
  int[] array = new int[n];
  int count = 0;
  
  void insert(int val) {
    if (count == array.length) {
      int sum = 0;
      for (int i = 0; i < array.length; ++i) {
        sum = sum + array[i];
      }
      array[0] = sum;
      count = 1;
    }
  
    array[count] = val;
    ++count;
  }

平均复杂度和均摊复杂度容易混淆，但均摊复杂度适用场景更小。对于上面示例这种大部分情况复杂度低，偶尔一次复杂度高，且它们之间出现有序时，才会使用均摊复杂度。

# 递归算法

递归就像梦中梦，从现实进入一层梦，继而二层梦，，再层层回来就，回到二层梦，再回到现实。

递归问题本质上是将一个大问题逐层分解为较小的子问题，这个分解问题的过程可以理解为递，直到递终止，然后由最里层逐层返回计算结果，直到最外层计算完毕，这个 结果逐层返回的过程，可以认为是归。

我们较为熟悉的递归问题有爬楼梯问题，斐波那契数列，系统菜单的展示算法，Java程序方法的递归调用本质上是方法栈帧的不断压栈，所以如果递归层次太深，可能会引起栈溢出的错误。 对于递归除了需要注意栈溢出，递归常常会存在重复计算。可以使用hash表将之前计算的结果存起来，以空间换区时间。

我们写递归常见的思维误区是人肉递归，常常发生脑容量溢出的奇效。因此面对一个递归，我们首先写好递归终止条件，然后写递归公式即可。

# 递归需要满足的三个条件

  • 一个问题的解可以分为几个子问题的解
  • 这个问题与分解之后的子问题，除了数据规模不同，求解思路完全一样
  • 存在递归终止条件

# 递归的优缺点及改写

优点是代码编写简洁。缺点是可能导致栈溢出，重复计算，函数调用耗时，空间复杂度高。 递归代码一般总是可以转换为非递归代码。递归本质上可以理解为循环，它的层可以转换为循环的次数，而每层的计算可以理解为循环计算的内容，只不过每层计算的结果 需要额外通过变量记录，以便下次循环使用。

# 排序算法

顺序表示一种秩序，通常是人们出于统计目的对一组值进行按照特定大小，先后规则进行调整，以达到特定维度观察事物，对未来进行预估的目的。比如对最近一周温度排序，以 查看最高温度和最低温度，对一组学生的成绩进行排序以确定第一名等。

# 如何分析排序算法

  • 排序算法的执行效率衡量点
    • 最好、最好、最坏情况时间复杂度
    • 时间复杂度的系数、常数、低阶
      • 当对同阶时间复杂度的排序算法性能比对时考虑
    • 比较次数、交换(或移动)次数
  • 排序算法的内存消耗
    • 原地排序：空间复杂度O(1)
  • 排序算法的稳定性 值相等的元素，经过算法的排序后，能保持其顺序的就是稳定算法
    • 订单按照金额从小到大排序，对于相同金额的订单按照下单时间从早到晚排序。
      • 可以按照时间从早到晚先排序，然后利用稳定的排序算法对金额从小到大排序。
  • 引入有序度和乱序度分析平均时间复杂度 有序度是具有有序关系的元素对个数 一组数1,2,3,4,5,6可以组合成(1,2),(1,3),.........15个序对 n*(n - 1) /2 如果全部正序，则有序度为15（满有序度），逆序度为0，如果全部逆序，则有序度为0，逆序度为15。 排序的过程可以理解为减少逆序度，增加逆序度到满有序度的过程。
  • 各排序算法时间复杂度、稳定性等总结
排序算法 原地排序算法 稳定排序算法 最好时间复杂度 最坏时间复杂度 平均时间复杂度

按算法时间复杂度大致分为3类

# O(n^2)

适合小规模数据的排序

  • 冒泡排序
    • 相邻元素进行大小比较，逆序则交换顺序
  • 插入排序
    • 在一个有序集合中，在能维持集合有序的位置插入新元素的思想
    • 分为已排序区间和未排序区间
      • 初始已排序区间里只有第一个元素
      • 从未排序区间里取出一个元素，插入到正确的位置保持已排序区间的有序。重复这个过程，直到未排序区间没有元素。
        • 在和已排序区间元素比较时，从头到尾和从尾到头比较，比较次数有区别
        • 整个排序过程移动总次数一定，等于逆序度，每次移动的个数等于每次增加的有序度数。
  • 选择排序
    • 类似插入排序，分为已排序和未排序区间，不同的是每次从未排序区间中通过比较选出最小的，放到已排序区间末尾。
    • 不是稳定排序算法，如3,3,5,1，经过第一次排序变成1,3,5,3，可以看到2个3的顺序已经变化了。

# O(nlogn)

适合大规模的数据排序

  • 归并排序（Merge Sort）
    • 使用分治思想
    • 将待排序数据从中间一分为二，再对划分后的每个子问题再进行划分，直到不可再分，之后对划分后的最小问题单元进行比较计算，得到一个个结果集，再将这些结果集一一合并。
    • 实现手段：递归
      递推公式：
merge_sort(p…r) = merge(merge_sort(p…q), merge_sort(q+1…r))
终止条件：
p >= r 不用再继续分解
      merge的方式是额外用一个临时数组存放merge结果，最后放入原数组。
    • 空间复杂度O(n)
  • 快速排序(QuickSort)
    • 分治、分区思想
    • 在pivot(基准)选择合理情况也就是最好时间复杂度O(nlogn)
    • 在pivot选择不合理情况也就是最坏时间复杂度O(n^2)

# O(n)

线性排序，非基于比较的排序算法

  • 桶排序
    • 待排序数据分到有限的有序桶里。
    • 单独对每个桶里的数据进行排序
    • 依序取出桶里的数据。
    • 时间复杂度为O(n) n个数，m个桶。每个桶k = n/m个元素，k个元素快排时间复杂度O(klogk)，每个桶总共复杂度O(mklogk)=O(nlogn/m)，当桶个数m接近n 时，时间复杂度接近于O(n)
    • 桶排序苛刻的使用场景
      • 待排序数据容易分成m个桶
      • 桶与桶之间有天然的大小顺序
      • 各个桶里的数据分布均匀，否则退化为O(nlogn)的时间复杂度。
    • 桶排序适合用在外部排序中，如对外部磁盘上大数据量无法一次加载到较小内存中进行排序的场景
      • 用100M内存对10G订单数据进行排序。
        • 扫描订单数据，确定金额范围如1~10w
        • 将金额划分为100个桶。对桶进行编号，桶依次存放1~1000,1001~2000区间金额的订单等
        • 将订单数据分别放入这100个桶，对应100个文件。分别对这100个文件进行排序。
        • 依次读入排好序的文件，将其写入同一个文件中，这个文件存放的就是排好序的订单数据。
  • 计数排序
    • 用在数据范围不大的场景中
    • 只能给非负整数排序，其他数据要转换为非负整数
  • 基数排序
    • 需要分割出独立的”位“来比较，位之间有递进关系，如a的高位比b大，剩下的低位就不用比较了。
    • 每一位的数据范围不能太大，要可以使用线性排序算法来排序，否则基数排序的时间复杂度无法做到O(n)
    • 对10w手机号排序，对单词进行排序（补齐0再排序）

# 如何实现一个通用的排序算法

O(n)时间复杂度的线性排序使用场景严格，因此不适用于做通用算法，O(n^2)时间复杂度的算法适用于小规模数据的排序。作为一个通用算法，选择O(nlogn)时间复杂度的算法 实现，在此基础上再结合数据规模做些优化，O(nlogn)时间复杂度的算法有归并、快排、堆排，但归并不是原地排序算法，空间复杂度高，不适合。因此一般采用归并排序作为 通用实现算法，归并是基于递归实现的，因此有栈溢出等风险，因此可以基于堆实现方法调用栈。脱离系统内置的栈限制。另外对于小规模数据的排序，O(n^2)时间复杂度的算法 实际占用时间可能小于O(nlogn)的，因此在小规模数据场景下，算法可以退化为使用O(nlogn)的插入排序算法。
