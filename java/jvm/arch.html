# 类加载子系统

# 加载

类加载器负责在运行期将类动态加载到jvm。它们是jre的一部分。(在oracle jdk中位于rt.jar下)。类加载器负责和底层文件系统交互，这样jvm就没必要和底层文件系统交互了。 java类是按需载入内存的，这也是类加载器大展身手的地方。

# 内置类加载器类型

  • bootstrap类加载器
    它是其他类加载器的父加载器，它是以本地代码编写的。所以System.out.println("Classloader of ArrayList:", ArrayList.class.getClassLoader());输出null。因为bootstrap类加载器不是java类表示的。 正因此，不同jvm的bootstrap类加载器行为可能会不同。
    • java类通过java.lang.ClassLoader实例加载，然而类加载器本身也是类，它们由bootstrap类加载器加载
    • 主要负责加载jdk内部类，通常是rt.jar和其他位于JAVA_HOME/jre/lib下的核心库。
  • sun.misc.Launcher$ExtClassLoader扩展类加载器
    • 是bootstrap类加载器的孩子。
    • 它加载标准核心java类的扩展类，加载JAVA_HOME/lib/ext目录或系统属性java.ext.dirs指定的目录下的类库
  • sun.misc.Launcher$AppClassLoader应用（系统）类加载器
    • 是扩展类加载器的孩子
    • 加载类路径中我们编写的类（应用程序级别的类），加载classpath环境变量或命令参数-cp或-classpath指定的类

# 类加载器如何工作的？

当jvm请求一个类时，类加载器将使用完全限定的类名尝试定位类，并将类定义加载到运行时。负责加载到运行时的方法是java.lang.ClassLoader#loadClass(java.lang.String)。如果类还未加载过，在父类加载器上调用 loadClass方法，依次调用父类的loadClass方法直到bootstrap类。然后从bootstrap类开始加载指定类，如果找不到指定类，再由bootstrap的子类调用java.net.URLClassLoader.findClass()，依次直到当前类加载器。 如果最终找不到会扔出ClassNotFoundException。

protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException
    {
        synchronized (getClassLoadingLock(name)) {
            // First, check if the class has already been loaded
            Class<?> c = findLoadedClass(name);
            if (c == null) {
                long t0 = System.nanoTime();
                try {
                    if (parent != null) {
                        c = parent.loadClass(name, false);
                    } else {
                        c = findBootstrapClassOrNull(name);
                    }
                } catch (ClassNotFoundException e) {
                    // ClassNotFoundException thrown if class not found
                    // from the non-null parent class loader
                }

                if (c == null) {
                    // If still not found, then invoke findClass in order
                    // to find the class.
                    long t1 = System.nanoTime();
                    c = findClass(name);

                    // this is the defining class loader; record the stats
                    sun.misc.PerfCounter.getParentDelegationTime().addTime(t1 - t0);
                    sun.misc.PerfCounter.getFindClassTime().addElapsedTimeFrom(t1);
                    sun.misc.PerfCounter.getFindClasses().increment();
                }
            }
            if (resolve) {
                resolveClass(c);
            }
            return c;
        }
    }

# 类加载器特性

  • 委派模型
    应用类加载器加载类或资源时，首先委托扩展类加载器加载，扩展类加载器再委托给bootstrap类加载器。只有当bootstrap类加载器和扩展类加载器不能加载到指定类时，应用类加载器才会自己去加载。
  • 类的唯一性
    向上委托父类查找的特性确保了类加载的唯一性。
  • 可见性
    子类加载器可以看到父类加载器加载的类。什么意思？
    应用类加载器加载的类可以看到扩展类及bootstrap类加载器加载的类，反之则不成立。如A是应用类加载器加载，B是扩展类加载器加载，那么应用类加载器加载的其他类（如C类）可以看到A和B。 但是扩展类加载器加载的D类只能看到B类。可以想象应用类加载器是最外面的圈圈，里面是扩展类加载器圈圈，再里面是bootstrap类加载器圈圈。最外面的圈圈可以看到里面圈圈加载的类，反之不然。

# 自定义类加载器

  • 使用场景

    • 修改字节码，如织入agents
    • 创建动态满足用户需求的类。比如，在JDBC里，在不同的驱动实现间切换是通过动态类加载完成的。
    • 为相同包名和类名的类加载不同的字节码时，实现类版本控制机制。这个可以通过URL类加载器（通过URLs加载jars）或自定义类加载器实现。
    • 浏览器，使用自定义类加载器从网站加载可执行内容。使用独立的类加载器从不同的网页加载applet。不同类加载器加载的相同名称applet被认为是不同的组件。

  • 实现自定义类加载器

    • 继承抽象类ClassLoader并实现findClass方法以遵守委派模型
public class CustomClassLoader extends ClassLoader {
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] b = loadClassFromFile(name);
        return defineClass(name, b, 0, b.length);
    }

    private byte[] loadClassFromFile(String fileName) {
        InputStream inputStream = getClass().getClassLoader().getResourceAsStream(
                fileName.replace('.', File.separatorChar) + ".class"
        );
        byte[] buffer;
        ByteArrayOutputStream byteStream = new ByteArrayOutputStream();
        int nextValue = 0;
        try {
            while ((nextValue = inputStream.read()) != -1) {
                byteStream.write(nextValue);
            }
        } catch (IOException e) {
            e.printStackTrace();
        }
        buffer = byteStream.toByteArray();
        return buffer;
    }
}

# 类加载器核心API

  • java.lang.ClassLoader#loadClass(java.lang.String, boolean)
    • 当父类加载器未能加载到指定类时，当前类加载器将调用findClass方法
  • java.lang.ClassLoader#findClass
    • java.net.URLClassLoader#findClass
    • 自定义类加载器实现findClass
    • 最终调用final ClassLoader#defineClass。该方法扔出ClassFormatError
  • java.lang.ClassLoader#getParent
    • 返回父类加载器
    • @CallerSensitive sun.reflect.Reflection.getCallerClass
  • java.lang.ClassLoader#getResource
    • 查找指定名称的资源(音频，图片，文本等)
    • 先委托父类查找资源，如果父类为null，则搜索内置到jvm的类加载器路径，如果都失败了，调用java.lang.ClassLoader#findResource查找。
      • java.net.URLClassLoader#findResource
      • 指定的资源名称可以是类路径的相对路径或绝对路径
    • Java从类路径上加载资源
    • 返回URL对象
    • Java中的资源加载与位置无关，只要设定了查找资源的环境，代码在哪运行无关紧要。

# 上下文类加载器 Context Classloaders

当遇到bootstrap类加载器加载的核心类需要使用应用类加载器加载的类时（即加载对应用类加载器可见的类），委派模型的类加载机制不能满足这个需求。如在JNDI核心功能是在rt.jar里的 bootstrap类实现的。但这些JNDI类可能需要加载独立厂商实现的JNDI提供程序（在应用程序类路径）。变通的方法是使用线程上下文类加载器。java.lang.Thread#getContextClassLoader， 若没设置contextCLassLoader，则默认为父线程的类加载器上下文。

  • 我的理解是JNDI核心类线程在调用应用代码线程时，暂时将应用代码的类加载器设置到核心类线程的上下文类加载器以加载应用资源 类似如下代码
// The previous context ClassLoader
final ClassLoader contextCL = Thread.currentThread().getContextClassLoader();
try {
    // Set my custom ClassLoader to make my classes accessible
    Thread.currentThread().setContextClassLoader(myCustomCL);
    // Execute some code here that will be able to access to classes or resources from
    // my specific folders and/or jar files

} finally {
    // Restore the previous CL
    Thread.currentThread().setContextClassLoader(contextCL);
}
